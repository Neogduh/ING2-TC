Guide Kubernetes

Kubernetes

Introduction

Kubernetes est un orchestrateur de containers. Il s’agit d’un système distribué, ce qui signifie qu’il peut fonctionner sur plusieurs machines.

Comme tout orchestrateur, il est composé d’un control plane et d’un data plane.

Le control plane correspond au « cerveau » du système. C’est avec lui que les opérateurs intéragissent pour configurer le cluster. Dans le cas de Kubernetes, le control plane est composé d’une base de donnée, un serveur API, un scheduler dont le rôle est d’assigner des workloads à des machines et quelques autres composants.

Le data plane correspond à la couche qui exécute les workloads, c’est-à-dire les containers dans notre cas. Elle est pilotée par le control plane.

Kubernetes se contrôle via une API REST. Habituellement, on utilise l’outil en ligne de commande kubectl pour communiquer avec cette API. Afin de gérer les ressources du cluster, on utilise souvent des fichiers YAML, appelés manifests, qui contiennent les déclarations des différents objets d’API.

Les objets Kubernetes

L’API Kubernetes propose de nombreux types de ressources afin de piloter le cluster. Voici une liste des objets essentiels à connaître pour débuter sur Kubernetes.

Namespace

Un namespace est une sorte de dossier dans lequel on va stocker d’autres objets. Cela permet d’organiser nos déploiements par exemple. Il est possible d’effectuer du contrôle d’accès sur les namespace ce qui permet par exemple de limiter l’accès d’une personne à un nombre restreint de namespaces.

Pod

Les Pods sont la ressource principale d’un cluster Kubernetes. Pour des raisons de simplicité de ce cours, un Pod est un container dans l’univers de Kubernetes.

Il est rare de créer des Pods à la main, en général on utilise d’autres types d’objets comme des Deployments ou des StatefulSet afin d’en générer.

Deployment

Un Deployment est un générateur de Pod. Les Deployments offrent plusieurs fonctionnalités quant à la génération des Pods :

  • possibilité de préciser un nombre de réplicas
  • mise à jour en RollingUpdate : lorsqu’on change sa configuration, un Deployment va créer un nouveau Pod avec la nouvelle configuration et attendre que ce Pod soit prêt avant de supprimer l’ancien

StatefulSet

Un StatefulSet est aussi un générateur de Pod. La différence avec un Deployment est que les Pods générés par un StatefulSet ont une identité unique. Les StatefulSet offrent la possibilité de générer des volumes persistants pour chaque Pod qu’il gère.

Un exemple d’utilisation de StatefulSet est pour les systèmes de base de donnée. Chaque réplica a besoin de son propre volume pour y stocker ses données.

Service

Afin de faire communiquer des Pods entre eux ou de les rendre accessibles à l’extérieur du cluster, on utilise des Services. Un service permet d’exposer un ou plusieurs ports d’un Pod ou d’un groupe de Pods. Lorsqu’un Service pointe sur plusieurs Pods, le Service répartit la charge entre tous ces Pods.

Dans le cadre de ce TP, nous n’utiliserons que des Services de type ClusterIP, c’est à dire des Services qui exposent des ports réseau à l’intérieur du cluster uniquement.

Les services ClusterIP sont accessibles aux autres Pods en utilisant la nomenclature DNS suivante : NOM_SERVICE.NOM_NAMESPACE.svc.cluster.local.

Les Service utilisent les labels des Pods pour déterminer à quel Pods le trafic doit être routé.

Ingress

Les Ingress sont des objets qui étendent le fonctionnement des services lorsqu’ils utilisent le protocole HTTP. Ils permettent de contrôler un reverse-proxy global au cluster.

ConfigMap

Les ConfigMap sont des objets simples qui permettent de stocker de la configuration sous forme de clé/valeurs. On peut utiliser un ConfigMap comme volume dans un Pod, il sera alors monté dans le container sous forme de dossier. Chaque configuration sera alors exposée dans un fichier ayant pour nom la clé de l’entrée. On peut aussi exposer les ConfigMap comme variables d’environnement dans des containers.

Secrets

Les Secrets sont des objets similaires aux ConfigMaps sauf qu’ils sont censés contenir des informations sensibles comme des mots de passe.

Installation de Kubernetes

Nous allons ici utiliser une distribution légère de Kubernetes appelée k3s. Elle contient tous les services nécessaires à faire tourner un cluster Kubernetes dans un unique binaire.

Vous pouvez suivre cette vidéo pour installer k3s dans votre VM.

Lancez la commande suivante sur votre VM :

curl -sfL https://get.k3s.io | sh -

DANGER

N’exécutez pas cette commande sur votre PC au risque d’endommager votre configuration système.

Pour intéragir avec Kubernetes, vous devez utiliser l’outil en ligne de commande kubectl sur votre PC. Cet outil parle avec le(s) serveur(s) primaire(s) Kubernetes via son API REST. Afin de vous y authentifier, récupérez le fichier /etc/rancher/k3s/k3s.yaml depuis la VM et placez le sur votre PC dans ~/.kube/config (attention, config doit être le nom du fichier, ce n’est pas un dossier). Editez ensuite le fichier pour remplacer l’occurrence localhost par l’IP de la VM.

Pour vérifier que votre configuration fonctionne bien :

$ kubectl cluster-info

CoreDNS is running at https://192.168.122.145:6443/api/v1/namespaces/kube-system/services/kube-dns:dns/proxy

Premier déploiement

Nous allons d’abord voir comment déployer une application simple dans son cluster Kubernetes.

En premier lieu, il nous faut un namespace. Nous avons ici deux options, soit on écrit un YAML qui le décrit, soit on utilise simplement kubectl.

$ kubectl create ns mon-namespace
namespace/mon-namespace created

ou alors

# namespace.yml
---
apiVersion: v1
kind: Namespace
metadata:
  name: mon-namespace
$ kubectl apply -f namespace.yml
namespace/mon-namespace created

Maintenant nous allons écrire un premier déploiement simple de nginx.

# deployment.yml
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
  labels:
    app: nginx
spec:
  replicas: 1
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
        - name: nginx
          image: nginx:1.25.3
          ports:
            - name: http
              containerPort: 80
$ kubectl apply -n mon-namespace -f deployment.yml
deployment.apps/nginx created
$ kubectl get pods -n mon-namespace
NAME                     READY   STATUS    RESTARTS   AGE
nginx-789bf7b8fc-w55pg   1/1     Running   0          2s

Maintenant, essayons de rendre accessible notre Pod au reste du cluster via un Service.

# service.yml
---
apiVersion: v1
kind: Service
metadata:
  name: nginx
spec:
  selector:
    app: nginx
  ports:
    - name: http
      port: 80
      targetPort: http
      protocol: TCP

Comme on ne spécifie pas de type, ce Service sera de type ClusterIP par défaut.

$ kubectl apply -n mon-namespace -f service.yml
service/nginx created
$ kubectl get svc -n mon-namespace
NAME    TYPE        CLUSTER-IP     EXTERNAL-IP   PORT(S)   AGE
nginx   ClusterIP   10.43.57.192   <none>        80/TCP    2s

Ce service sera accessible aux autres pods du cluster à l’aide du nom de domaine nginx.mon-namespace.svc.cluster.local.

Nous allons maintenant rendre notre service accessible à l’extérieur du cluster. Comme il s’agit d’un service HTTP, nous allons utiliser un objet de type Ingress afin d’utiliser le reverse proxy du cluster.

# ingress.yml
---
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: nginx
spec:
  rules:
    - http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: nginx
                port:
                  name: http
$ kubectl apply -n mon-namespace -f ingress.yml
ingress.networking.k8s.io/nginx created
