3. Docker Compose

Contexte

Vous travaillez dans la toute nouvelle et toute petite entreprise Sundaland qui se lance sur le marché florissant de l’e-commerce en créant sa propre plateforme d’achats en ligne. Elle a donc développé en interne une application web qui permet à des clients de faire des achats. Comme l’entreprise est jeune, l’équipe dev n’est composée que d’une personne. L’application dispose d’un backend sous forme de monolithe développé en Java avec Quarkus et un frontend en Javascript avec Vue.

Le développeur étant parvenu à une version suffisamment fonctionnelle pour être mise en production, le fondateur et CEO de la startup vous missionne pour rendre sa plateforme accessible au grand public afin d’effectuer ses premières ventes au plus vite.

Tout le code de l’application est stocké dans un dépôt GitLab sur lequel vous êtes ajouté. Vous n’êtes cependant pas habilité à modifier le code. Lors de votre déploiement, le code doit donc rester intact.

Introduction

Le déploiement à la main fonctionne mais pose un certain nombre de problèmes qui deviennent de plus en plus dérangeants lorsqu’on passe à l’échelle :

  • La mise à jour de l’application nécessite beaucoup d’interventions manuelles :
    • Compilation
    • Transfert des fichiers vers le serveur
    • Redémarrage à la main dans le cas du backend
  • La compilation en local implique que vous ayez exactement les mêmes dépendances sur votre PC que sur le serveur de production.
  • Une mise à jour du système peut changer la version d’une dépendance (MongoDB ou Java par exemple) qui apporte des breaking changes rendant l’application inopérante.
  • Une réinstallation sur une autre machine nécessite de recommencer toutes les étapes du déploiement ce qui est chronophage et fastidieux.
  • D’un point de vue organisationnel le contrôle d’accès est impossible : si on veut qu’un développeur puisse déployer en production, il doit avoir un accès total à la machine. Cela pourrait poser problème si la machine contient des données sensibles qu’un développeur ne doit pas voir.

Dans cette partie, nous allons nous attaquer aux 3 premiers points. En utilisant Docker et Docker Compose :

  • Les mises à jour système n’impactent plus les dépendances des applications car ces dernières sont bundle dans les images. C’est aux développeurs d’assurer que les images Docker contiennent les bonnes dépendances pour que leur application fonctionne.
  • Tout se compile via les Dockerfile, plus besoin de faire attention aux dépendances qu’on installe sur son PC. On peut même imaginer ajouter une CI qui build les images Docker automatiquement.
  • La mise à jour est rendue plus simple, on doit juste changer le tag de l’image.
  • La réinstallation de l’application implique juste d’installer Docker et de lancer le Docker-Compose.

Contexte

L’équipe dev de l’entreprise s’agrandit et une nouvelle recrue prend l’initiative de créer des images Docker pour le backend et le frontend. Comme il fait bien les choses, il a mis en place une CI sur le dépôt GitLab pour les construire automatiquement et les publier dans le registre.

Votre travail

Votre but va maintenant être de redéployer l’application en utilisant uniquement Docker Compose. Vous ne devez plus faire tourner un seul service lié à l’application en dehors de containers.

Votre Docker Compose devra donc contenir les services suivants :

  • Backend sundaland-backend
  • Frontend sundaland-frontend
  • MongoDB mongodb
  • Nginx nginx

Voici les images Docker à utiliser :

  • registry.cri.epita.fr/ing/majeures/tc/info/sundaland/sundaland-backend:1.0.0
  • registry.cri.epita.fr/ing/majeures/tc/info/sundaland/sundaland-frontend:1.1.1
  • registry.cri.epita.fr/ing/majeures/tc/info/sundaland/mongo:7.0.5
  • registry.cri.epita.fr/ing/majeures/tc/info/sundaland/nginx:1.25.3

Le container mongodb doit avoir un volume associé mongodb-data afin de persister les volumes de la base de données.

Les containers qui n’ont pas besoin de communiquer entre eux doivent être isolés à l’aide de réseaux Docker différents. Les containers du backend et de MongoDB doivent être dans un réseau sunda-back-net. Les containers nginx, frontend et backend doivent être dans un réseau sunda-net.

Le nom du projet Docker Compose doit être sundaland-prod.

Vous devez gérer la dépendance entre les services.

Vos services doivent redémarrer automatiquement.

Pour le container nginx, il est attendu que vous overridez la config /etc/nginx/conf.d/default.conf. Votre serveur doit écouter sur le port 8080 du container mais doit être exposé sur le port 80 de la machine (comme indiqué sur le schéma).

Vous ne devez pas spécifier la version de Docker Compose dans votre fichier.

ℹ️ Les images Docker ne sont pas accessibles publiquement. Vous devez générer un token d’accès personnel sur GitLab afin d’authentifier votre VM pour pull. Voici les étapes à suivre :

  • Créer un token d’accès personnel avec le scope read_registry
  • Taper la commande docker login registry.cri.epita.fr et utilisez votre token
  • comme mot de passe
