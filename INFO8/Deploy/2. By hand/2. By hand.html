2. By hand

Contexte

Vous travaillez dans la toute nouvelle et toute petite entreprise Sundaland qui se lance sur le marché florissant de l’e-commerce en créant sa propre plateforme d’achats en ligne. Elle a donc développé en interne une application web qui permet à des clients de faire des achats. Comme l’entreprise est jeune, l’équipe dev n’est composée que d’une personne. L’application dispose d’un backend sous forme de monolithe développé en Java avec Quarkus et un frontend en Javascript avec Vue.

Le développeur étant parvenu à une version suffisamment fonctionnelle pour être mise en production, le fondateur et CEO de la startup vous missionne pour rendre sa plateforme accessible au grand public afin d’effectuer ses premières ventes au plus vite.

Tout le code de l’application est stocké dans un dépôt GitLab sur lequel vous êtes ajouté. Vous n’êtes cependant pas habilité à modifier le code. Lors de votre déploiement, le code doit donc rester intact.

Introduction

À ce stade, comme l’application est flambant neuve et que rien n’est encore en production, il vous est demandé de faire au plus simple et de déployer l’application à la main. Un serveur virtuel vous est fourni, la quantité de ressources n’est pas énorme mais l’entreprise n’a pas encore le budget pour héberger son application sur une VM haute performance.

ℹ️ Dans cette situation, votre machine virtuelle pourrait être située sur un serveur localisé dans les locaux de l’entreprise (on-premise), dans un datacenter ou directement fournie par un hébergeur (exemple : OVH). Aujourd’hui, les offres d’hébergement ont évolué et des providers clouds sont apparus. Votre machine virtuelle pourrait donc être une instance EC2 s’il s’agit d’Amazon Web Service par exemple.

⚠️ Dans le cadre du TP, nous n’avons pas de cloud provider à disposition car cela a un coût, que ce soit pour un cloud public ou sur le cloud privé de l’école. Notre capacité serveur est insuffisante pour héberger les machines virtuelles de tous les étudiants qui suivent ce cours. Vous allez donc devoir créer votre VM sur votre machine (soit votre PC perso soit en salle machine). Si vous utilisez Linux, je vous recommande fortement d’utiliser libvirt et son interface graphique virt-manager. Sur Windows ou macOS, vous pouvez utiliser VMWare. Attention, VirtualBox ne fonctionnera pas car il manque des instructions CPU pour MongoDB.

Sur les machines de l’école, merci de prendre connaissance du guide libvirt (disponible sur l’intranet).

Architecture finale

Installation de la VM

  1. Créez une machine virtuelle Ubuntu Server 22.04 avec les spécifications suivantes :

    Système : Ubuntu 22.04
Nombre de cœurs de CPU : 4
Mémoire vive : 4096 Mo

  2. Suivez les étapes d’installation d’Ubuntu Server. La configuration par défaut suffit pour tout. Pensez à bien activer le serveur OpenSSH lorsque c’est demandé.

    Hostname : sundaland-prod

  3. Lorsque l’installation est terminée, il vous est proposé de reboot la VM. Redémarrez donc votre machine virtuelle. Il vous sera demandé de retirer le CD d’installation, retirez donc le lien vers le fichier ISO dans les paramètres de la VM et appuyez sur Entrée dans la console de la VM.

  4. Après que la VM est démarrée, connectez vous via la console. Utilisez ensuite la commande ip -c addr pour récupérer l’adresse IP de la VM. Déconnectez vous ensuite.

⚠️ À partir de maintenant, vous ne devez plus utiliser la console de la VM pour la configurer. Connectez vous en SSH à la machine virtuelle en utilisant l’IP que vous avez noté.

Déploiement des dépendances et de l’application

L’objectif est donc de déployer l’application Sundaland sur la VM. À la fin du déploiement, vous devez pouvoir accéder au site via le navigateur web de votre PC en utilisant l’adresse IP de votre VM et en ne précisant aucun port.

Installation de MongoDB

Suivez les instructions d’installation de MongoDB sur Ubuntu 22.04.

  • Ajouter le dépôt APT
  • Installer le package mongodb-org
  • Activer et démarrer le service mongod

Préparation de MongoDB

  • Avec la commande mongosh, créer la base de donnée Sundaland
  • Créer un utilisateur sundaland-backend avec le mot de passe de votre choix dans la base de donnée Sundaland avec droits de lecture et écriture sur celle-ci
  • Créer les collections Products et Purchases

Installation des dépendances du backend

Vous devez installer les dépendances nécessaires au runtime du backend. Comme il s’agit d’un serveur de production, il n’y a pas besoin d’installer les dépendances nécessaires pour le développement. Contentez-vous donc d’installer le package openjdk-17-jre-headless.

Installation du backend

Vous devez transférer le jar du backend sur le serveur dans le dossier /srv/sundaland. Pour simplifier le déploiement, vous devez générer un über-jar du backend, c’est-à-dire un jar qui contient toutes les bibliothèques qui ont été utilisées par les développeurs de l’application ce qui vous évite d’avoir à les installer vous même. Référez-vous au README du backend pour générer le fichier jar.

Configuration du backend

Configurez le backend dans le fichier /srv/sundaland/.env. Référez-vous au README du backend pour avoir la liste des variables d’environnement disponibles.

Test du backend

À ce stade, vous devriez être en mesure de lancer le backend à la main. Celui-ci sera accessible au lien suivant : http://IP_VM:8080. Comme le backend est une simple API REST, vous aurez une erreur 404 en ouvrant ce lien. Vous pouvez cependant accéder au Swagger en utilisant la route /q/swagger-ui.

Afin de tester le bon fonctionnement de la connexion avec MongoDB, utilisez la route /api/products.

⚠️ L’accès au backend via le port 8080 ne doit être utilisé que pour des fins de test. À la fin du déploiement, le frontend ET le backend devront être accessibles via le port 80. Pour cela, on utilisera un reverse-proxy, il ne faut donc pas modifier le port d’écoute du backend.

Pour fermer le backend, utilisez le racourci CTRL+C.

Installation du reverse-proxy

Pour pouvoir accéder au frontend et au backend sur le même port, nous allons utiliser un reverse-proxy. Son rôle sera de router le trafic de la route /api vers le backend et le reste vers le frontend. Nous allons ici utiliser nginx qui est un des serveur web les plus utilisé au monde aujourd’hui.

Installez donc nginx via apt.

Il sera normalement démarré automatiquement via systemd.

Installation du frontend

Vous allez en premier lieu avoir besoin de build le frontend sur votre PC. Référez vous au README du frontend. Avant de build, pensez à bien changer le fichier .env.production pour qu’il contienne l’IP de votre VM. Il ne faut pas spécifier de port dans ce fichier, sinon c’est de la triche.

À la fin du build, vous disposerez d’un dossier dist qui contient les fichiers à placer sur le serveur.

Supprimez le fichier déjà dans /var/www/html et copiez-y les fichiers du dossier dist.

À ce stade, le frontend sera accessible sur http://IP_VM. Cependant, il ne peut pas parler au backend.

Configuration de nginx

Pour que le backend soit accessible sur le même port que le frontend, vous devez modifier la configuration de nginx présente dans /etc/nginx/sites-enabled/default.

Ajoutez la location /api pour qu’elle soit proxy vers http://localhost:8080.

Il faudra redémarrer nginx via systemctl pour que les changements de configuration soient appliqués.

Test

À ce stade, vous devriez avoir un déploiement fonctionnel. Cependant, vous faites face à plusieurs problèmes :

  • Il faut lancer le backend à la main via SSH.
  • Vous ne pouvez pas quitter votre session SSH, sinon le backend se ferme (ou alors il faut utiliser un multiplexeur de terminaux, c’est pas optimal).
  • Le backend a tendance à crash ce qui nécessite votre intervention pour le relancer.
  • Un redémarrage de la VM nécessite votre intervention pour redémarrer le backend.

Pour palier ces problèmes, on peut utiliser le gestionnaire de service du système, c’est à dire systemd. Celui-ci pourra démarrer le backend au démarrage et le redémarrer en cas de problème.

À la fin de cette étape, vous pourrez redémarrer votre VM et l’application sera accessible sans intervention de votre part.

Écrire le service sundaland-backend.service

Contraintes :

  • Doit start après que le réseau soit en ligne (il existe un .target pour ça) et mongod.service
  • Doit charger le fichier d’environnement /srv/sundaland/.env
  • Doit avoir comme working directory /srv/sundaland
  • Doit restart en cas d’échec de l’application

Activez et démarrez ce service. N’oubliez pas de recharger les services systemd lorsque vous faites des modifications sur votre fichier de service.

Rendu

Votre fichier de service systemd est à rendre via l’activité sur l’intranet Forge.

⚠️ La moulinette parse le fichier pour effectuer quelques tests dessus, cependant elle n’est pas capable d’identifier certaines syntaxes normalement valides dans systemd.

Dans une unit systemd, si une option a plusieurs valeurs, il est possible soit de la faire apparaître plusieurs fois, soit de la faire apparaître qu’une fois et de séparer les valeurs par un espace. Dans la moulinette, seul le deuxième cas est possible. Une option qui a plusieurs valeurs doit donc apparaître une unique fois et les valeurs doivent être séparées par un espace sur la même ligne.

Exemple : MonOption=valeur1 valeur2

Ce fichier est l’unique rendu de ce TP 1. Vous pourriez être tenté de ne pas le suivre et simplement écrire ce fichier mais cela n’aurait aucun intérêt et je vous déconseille fortement de procéder de cette manière.
